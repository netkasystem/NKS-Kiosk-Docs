1. System Overview

ระบบลงทะเบียนและจัดการความยินยอม (Consent Management) ผ่านตู้ Kiosk แบบ Self-Service สำหรับ Maersk โดยผู้ใช้สอดบัตรประชาชน ถ่ายภาพใบหน้า และให้ความยินยอม (Consent) ตามกฎหมาย PDPA ก่อนส่งข้อมูลไปยังระบบ Netka Intelligent

Main Flow (New User)

Welcome → Privacy Notice → Insert Card → Read Card → Face Capture → Confirm Data → Consent Form → Send to Netka Intelligent → Complete

Returning User Flow

Welcome → Insert Card → Read Card → Show Previous Consent → Edit & Re-submit → Send to Netka Intelligent → Complete

PDPA Compliance: ระบบรองรับ Granular Consent — แยกความยินยอมเป็นรายข้อ ผู้ใช้เลือกยินยอมหรือไม่ยินยอมแต่ละข้อได้อิสระ พร้อมบันทึก log ครบถ้วน (ใคร, อะไร, เมื่อไหร่, ผ่านช่องทางใด)

2. Actors & Systems

Actor / System Description
Visitor (ผู้ใช้บริการ) ลงทะเบียนและให้ความยินยอมที่หน้าตู้ Kiosk
Kiosk Terminal แสดง UI, สั่ง Card Reader, ถ่ายภาพใบหน้า
Card Reader (Hardware) อ่านข้อมูลจากชิปบัตรประชาชน ผ่าน SignalR Hub
Camera / CCTV (Hardware) ถ่ายภาพใบหน้าสด (Live Photo) พร้อม Face Detection
N-Kiosk Web Server ประมวลผล, จัดเก็บข้อมูล, จัดการ Consent (ASP.NET Core)
File Server (nginx) จัดเก็บไฟล์รูปภาพ (รูปบัตร, รูปใบหน้า)
Netka Intelligent (External) ระบบปลายทางที่รับข้อมูลลงทะเบียน, ภาพถ่าย, และ Consent
NDPP Server (External) ระบบจัดการ Privacy Consent ส่วนกลาง

3. Preconditions

  • ตู้ Kiosk เปิดอยู่และเชื่อมต่อ Web Server ได้
  • Card Reader เชื่อมต่อผ่าน SignalR Hub สำเร็จ
  • กล้อง / CCTV พร้อมใช้งาน
  • ผู้ใช้มีบัตรประชาชนแบบ Smart Card
  • File Server (nginx) พร้อมรับ upload
  • Netka Intelligent API พร้อมรับข้อมูล

5.2 Privacy Notice — แจ้งเตือนสิทธิ

แจ้งเตือนสิทธิ
Privacy Notice

ตู้นี้มีการเก็บข้อมูลส่วนบุคคลและภาพถ่ายเพื่อใช้ในการลงทะเบียนและยืนยันตัวตน

ข้อมูลที่จัดเก็บ:

  • ข้อมูลจากบัตรประชาชน
  • ภาพถ่ายใบหน้าจากกล้อง

ข้อมูลจะถูกส่งไปยังระบบ Netka Intelligent เพื่อดำเนินการตามวัตถุประสงค์ที่แจ้งไว้

รับทราบ ดำเนินการต่อ
ยกเลิก
Description: แจ้งให้ผู้ใช้ทราบก่อนเริ่มกระบวนการว่าตู้นี้มีการเก็บข้อมูลอะไรบ้าง
Data Source: ดึง Privacy Notice จาก NDPP ผ่าน GetPrivacyNotice proxy API
  - เรียก GetPolicyOuByKey ด้วย key จาก variable kiosk_ndpp_privacy_notice
  - แสดง HTML content จาก NDPP / ถ้าไม่มีจะแสดง fallback content เดิม
  - แสดง Loading spinner ระหว่างรอ API → disable checkbox จนกว่าจะโหลดเสร็จ
PDPA Requirement: ต้องแจ้งให้ทราบก่อนเก็บข้อมูล (Privacy Notice)
Action: กด "รับทราบ" เพื่อดำเนินการต่อ
Navigation: รับทราบ → Insert Card / ยกเลิก → Welcome
Notes: หน้านี้แยกจาก Consent Form — เป็นการแจ้งให้ทราบ ไม่ใช่การขอความยินยอม

ยืนยันรายการที่เลือก
ยินยอมทั้งหมด
Description: แบบฟอร์มความยินยอมแบบ Granular Consent (แยกรายข้อ)
PDPA Requirement:

  - ผู้ใช้เลือกยินยอม/ไม่ยินยอมแต่ละข้อได้อิสระ

  [ ] บันทึกภาพใบหน้าและข้อมูลจากกล้อง CCTV

Dynamic: รายการ consent สามารถ config ได้จาก NDPP server
On Submit:

  2. ส่งข้อมูล + ภาพถ่าย + Consent ไป Netka Intelligent

6. Data Flow & Integration

6.1 System Integration Diagram

From To Protocol Data
Card Reader Kiosk Browser SignalR (WebSocket) ข้อมูลบัตร + รูปจากชิป
Camera / CCTV Kiosk Browser WebRTC (getUserMedia) Live Photo ใบหน้า (base64)
Kiosk Browser N-Kiosk Web Server HTTPS + AES-ECB Encrypted payload (card data, face, consent)
Web Server Database (SQL) Entity Framework Enrollment, Consent Log
Web Server File Server (nginx) HTTP PUT (WebDAV) photo.jpg, face.jpg
Web Server NDPP Server HTTP PUT + JWT Consent form data (purpose options)
Web Server Netka Intelligent HTTP POST ข้อมูลบัตร + ภาพถ่าย + Consent + Kiosk ID + Timestamp

8. Returning User Flow

Requirement: หากผู้ที่เคยลงทะเบียนแล้ว เวลาเสียบบัตรเข้าไป ให้ขึ้นหน้า Consent ที่เคยให้ความยินยอมไว้ แล้วสามารถแก้ไขและส่งกลับไปอีกครั้งได้

8.1 Detection (Local DB)

หลังจากอ่านบัตรสำเร็จ (Step5) ระบบเรียก CheckReturningUser API โดยส่ง เลขบัตรประชาชน (idcard) เพื่อค้นหาใน table kiosk_consented:

API: GET /api/KioskApi/CheckReturningUser?idcard=XXX

Response Field Type Description
isReturning boolean พบเลขบัตรใน kiosk_consented หรือไม่
consentedId uint ID ของ record ล่าสุด (เมื่อ isReturning = true)
consentedDate datetime วันที่ consent ล่าสุด

Flow Summary

  • isReturning = true → Returning User Flow:
    1. Step5: Preload NDPP data (GetIntegrateNdpp, GetIntegrateNdppByKiosk, GetNdppPreferenceConsent)
    2. Step8: ข้าม Face Capture (Step9-11) ไปตรง Step17
    3. Step17: Auto-submit consent ทุก service (select all purposes) → NDPP + Netka Intelligent → StepEnd
  • isReturning = false → New User Flow (Step8 → Face Capture → Confirm → Consent → Complete)
หมายเหตุ: Returning User detection ใช้ เลขบัตรประชาชน (idcard) เช็คจาก kiosk_consented table ใน local DB
สำหรับ NDPP ยังมี API GetCheckDsExistConsentForm ที่ใช้ ชื่อ-นามสกุล ผ่าน FindDS() ซึ่งสามารถใช้เสริมได้ในอนาคต

8.2 Auto Re-submit (Returning User)

Returning User ไม่ต้องกรอกหรือแก้ไข consent — ระบบจะ auto-submit ทั้งหมด:

Step System Behavior
1. ดึง Consent Form เรียก GetIntegrateNdppForm สำหรับทุก service
2. Select All Purposes เลือก purpose_option ทั้งหมดโดยอัตโนมัติ
3. Submit เรียก SubmitIntegrateNdpp → ส่ง consent ไป NDPP + Callback ไป Netka Intelligent
4. Complete ไป StepEnd

9. Exception Handling

Scenario Detection Handling Navigation
บัตรอ่านไม่ได้ / ชำรุด Card Reader return error แจ้งเตือน → ลองใหม่ หรือ ยกเลิก Retry → Insert Card / Cancel → Welcome
บัตรหมดอายุ expiry_date < today แจ้งเตือน → ยกเลิก Welcome
ไม่สอดบัตรภายใน timeout SignalR timeout auto-reset Welcome
Face detection ล้มเหลว face-api.js no detection ผู้ใช้กดถ่ายใหม่ Face Capture (retry)
File upload ล้มเหลว HTTP non-2xx from File Server Log error, continue flow ดำเนินการต่อ
Netka Intelligent API ล้มเหลว HTTP non-2xx / timeout Retry + แจ้ง error แสดง error, retry
NDPP API ล้มเหลว HTTP non-2xx Log error + แจ้ง แสดง error
ผู้ใช้ idle idle timer expired auto-reset Welcome
Network disconnect fetch / SignalR failure แสดง error, retry แต่ละหน้า
